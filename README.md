# 港险条款问答 RAG 评测基准集

从 2052 个企微真实对话出发，经 P0-P6 六阶段管道，生成一份严格符合 schema.json 的 RAG 评测数据集。

---

## 这个项目是干什么的？

**一句话**：给"AI 保险问答机器人"出一份考卷，考它到底能不能回答好用户的真实问题。

考卷不是凭空编的——它来自 2052 段真实的保险代理人和客户的聊天记录。每道题都标注了"标准答案在哪个文件的第几页"和"AI 正确的回答应该长什么样"。拿到这份考卷，你就可以客观地给任何一版 RAG 系统打分。

## 核心概念

### 评测集是什么、怎么用

一份评测集由 150-200 条"样本"组成。每条样本包含：

```
问题: 用户原始提问
标准证据: 答案在文件 b8062eeb... 的第 1 页
标准答案: AI 应该回答"保证利息 10%，2 年缴付期，美元保单"
不能出现: "6%" ← 那是 3 年缴的数字，答这个就错了
```

评测的时候，把你的 RAG 系统接上来，让它逐条回答。然后对着标准答案打分：事实对了没？关键词命中了没？有没有编造不该出现的内容？

### schema.json 的六层字段

每条样本有 6 层共 34 个字段：

| 层 | 作用 | 举例 |
|---|---|---|
| A 样本管理 | 这条能不能用、归哪类 | `sample_type`, `is_evaluable` |
| B 输入层 | 用户原话 + 问题特征 | `question_raw`, `question_quality` |
| C 检索层 | 答案在库里哪个位置 | `gold_evidence`(文件 hash + 页码) |
| D 生成层 | AI 该答什么 | `base_answer`, `forbidden_content` |
| E 评分配置 | 怎么打分 | `eval_method`, `score_weights` |
| F 标注元信息 | 谁标的、核验没 | `annotator`, `verified` |

## P0-P6 管线详解

### 🔧 P0 — 对话切分

**做什么**：把一段多轮聊天记录切成独立的问题单元。

**输入**：2052 个 session，每个是一段完整的聊天记录（user 和 assistant 交替对话）

**输出**：2495 个 question unit（问题单元）

**通俗理解**：

一段真实的企微聊天长这样：
```
用户: 在吗
AI:   你好呀，我是小飞飞~
用户: 我想问下
用户: 就是我买嗰份宏利
用户: 等待期几耐
AI:   宏挚传承的等待期是 90 天...
用户: 咁医疗果part呢
AI:   ...
```

这里有 6 条用户消息，但显然不是 6 个独立问题。P0 做的事：

- **拼起来**："我想问下"+"就是我买嗰份宏利"+"等待期几耐" — 这三条是用户一口气发的，属于同一个问题，拼成一句话
- **切开**：AI 回复之后用户说的"咁医疗果part呢" — 新问题开始了，切成另一个单元
- **标记打招呼**：第一条"在吗"标为寒暄，不进入正式评测

**P0 的一个重要原则**：只负责"把散的拼起来"，不负责"把缺的补上"。

```
❌ P0 不做: "那5年缴呢" → 补全成"宏挚传承5年缴的保费是多少"
✅ P0 做:   "那5年缴呢" 原样保留 + 在旁边标记"这问题可能是追问宏挚传承"
```

问题原文一字不改，上下文信息单独存在 `context_company` 和 `context_product` 字段里。

**最终结果**：

```
2495 个 question unit
├── 2220 个 有实质问题（进后续处理）
└──  275 个 纯寒暄（"在吗""谢谢"，不进评测）
```

### 🏷️ P1 — 五分类分流

**做什么**：给每个问题打上"这是什么类型"的标签。

**输入**：P0 切出的 2495 个 question unit

**输出**：2495 条，每条带 `sample_type` 标签

**通俗理解**：

2495 个问题里，不是每个都能进评测的。P1 像一个分拣员，把问题分成五类：

| 标签 | 含义 | 例子 | 进评测？ |
|---|---|---|---|
| `system_error` | AI 自己崩了 | Python traceback、系统繁忙 | ❌ 不进 |
| `chitchat` | 打招呼/闲聊 | "在吗""你好""谢谢" | ❌ 不进 |
| `multi_doc` | 跨保司比较 | "宏利和友邦哪个好" | ✅ |
| `correct_refusal` | AI 说找不到 | "抱歉无法找到相关信息" | ⚠️ 待 P4 核实 |
| `normal_answer` | 正常问答 | 问了、答了、有具体内容 | ✅ |

**P1 做的一个关键判断**：遇到 AI 说"找不到"，P1 不直接标 `correct_refusal`，而是标一个"待核实"标记。因为有一种常见情况——库里其实有答案，只是 AI 没搜到。

> 例子：用户问"预存利率是多少"，AI 说"找不到"。但去库里一看，"预存利率"在条款里叫"保证利息"。答案是 10%。这种就是 TERM-0007 案例——术语没对齐导致的假拒答。

这种样本在 P4 会被翻转。

**最终结果**：

```
1879  normal_answer
 239  multi_doc（跨保司比较）
  81  correct_refusal（待核实）
 246  chitchat（打招呼）
  50  system_error（系统崩了）
```

### 🎯 P2 — 分层抽样

**做什么**：从 2495 个已分类的问题中，按比例抽取 160 条候选样本。

**输入**：P1 的分类结果

**输出**：160 条候选，格式对齐 schema.json

**通俗理解**：

2495 条问题，不可能全部人工标注（成本太高）。但也不能闭眼随机抽——万一友邦抽了 30 条，安达人寿 1 条都没有，评测结论就不公平。

P2 像一个"公平的抽签官"，按三个维度确保样本代表性：

- **保司**：21 家保险公司每家至少 2 条（大公司可以多，小公司不能挂零）
- **问题类型**："优惠咨询""产品细节""理赔流程"各类型均匀分布
- **时间**：跨月份均匀分布，不集中在某个月

**各类配额**：

```
normal_answer:   1879 条池 → 抽 85 条  (主力)
correct_refusal:   81 条池 → 取 35 条  (全部进候选，等 P4 核实)
multi_doc:        239 条池 → 抽 25 条  (跨保司题)
chitchat:         246 条池 → 抽  8 条  (少量保留，验证过滤能力)
system_error:      50 条池 → 抽  7 条  (少量保留，验证过滤能力)
                                  ---
                          总计 160 条
```

**编号规则**：用纯流水号 `EVAL-0001` ~ `EVAL-0160`，不在编号里写类型。原因是：一条 `correct_refusal` 可能在 P4 被发现库里其实有答案，翻转为 `normal_answer`。如果编号叫 `REF-0001` 但内容已经变成 `normal_answer`，就尴尬了。

**P2 还做一件事**：把 160 条样本按 schema.json 格式搭好框架。A 层（管理字段）、B 层（输入层）自动填充，C 层（检索）、D 层（生成）留空 — 等着 P4 和 P5 来填。

### 🔍 P3 — 问题质量标注

**做什么**：自动检测问题文本的特征，生成标注指引，帮标注员完成 `question_quality` 的 7 个判定。

**输入**：P2 的 160 条候选

**输出**：160 条样本 + 自动检测提示 + 标注指引文档

**通俗理解**：

用户问的问题，本身就有"脏"特征。比如：

| 特征 | 例子 | 为什么重要 |
|---|---|---|
| 口语/粤语 | "等待期几耐" | RAG 检索可能搜不到 |
| 术语不对齐 | "预存利率"← 条款叫"保证利息" | RAG 最大失败原因 |
| 错别字 | "宏利"写成"红利" | 检索召回错误文档 |
| 信息不全 | "保费多少"没说是哪款产品 | 模型需要反问澄清 |
| 一句多问 | "保额多少？可以退保吗？" | 需要分点回答 |

P3 自动检测能确定的（粤语特征词、已知错别字、术语映射提示），不能确定的留给人判断。每条样本附带一份"小抄"：

```json
{
  "_p3_auto_hints": {
    "term_mapping_hints": [
      { "spoken_term": "受保人", "formal_term": "被保险人" }
    ],
    "typo_hints": [],
    "possible_multi_intent": null    // 拿不准，标注员自己看
  }
}
```

标注员对着小抄一条条复核 7 个 bool 值，2 分钟能标完一条。没有小抄的话，5-10 分钟一条。

**P3 还生成了** `p3_annotation_guide.md` — 标注员的操作手册，含每个字段的判定标准、示例、术语映射参考表。

### 📚 P4 — 检索核实（最关键的阶段）

**做什么**：去 DashVector 向量库里搜每个问题，让人挑出真正包含答案的文档，填 `gold_evidence`。

**输入**：P3 的 145 条可评测样本 + DashVector 向量库

**输出**：每条样本的 `gold_evidence`（答案在哪个文件第几页）+ 翻转误判的 `sample_type`

**通俗理解**：

前面几步都在处理"问题本身"。P4 回答的是评测集的核心问题：**对于这个问题，库里到底有没有答案？答案在哪？**

P4 做三件事：

**第一件：重查向量库。** 用 `question_raw` 直接去当前版本 DashVector 搜，返回 top-15 个最相关的文档片段。不信任系统当时返回的结果——系统当时用的是旧库，现在库已经重建过了。

**第二件：人工逐条选。** 标注员收到一份"核实工作单"：每个问题下面挂着 15 个检索结果（文件名、页码、分数、文本预览）。标注员逐条判断：

```
chunk 1 (score=0.78): "宏挚传承2年缴付期，保证利息10%..."
  → ✅ 这是答案，勾选进 gold_evidence

chunk 2 (score=0.56): "宏挚传承5年缴付期，保费折扣5-12%..."
  → ❌ 不相关（5年缴，不是2年缴），但分数高会误导模型
  → 标记为 irrelevant_high_score，传给 P5 提炼 forbidden_content
```

**第三件：翻转误判。** P1 标了 35 条 `correct_refusal`（AI 当时说"找不到"）。P4 一查库：

```
查到了 → 系统当时拒答是错的 → 翻转为 normal_answer
没查到 → 系统拒答是对的 → 确认为 correct_refusal + no_answer_in_corpus=true
```

> TERM-0007 就是翻转的经典案例："预存利率"库里其实有答案（10% 保证利息），系统没搜到不是因为没答案，是因为术语没对齐。不能标 correct_refusal，必须翻回 normal_answer。

**当前状态**：P4 脚本框架已就绪，DashVector 连接代码需要配置环境变量后启用。

### ✏️ P5 — 答案标注

**做什么**：标注每条样本"AI 应该答出什么"和"AI 一定不能答出什么"。

**输入**：P4 核实后的样本（含 gold_evidence）

**输出**：每条样本填写了标准答案字段

**通俗理解**：

P4 确定了"答案在哪个文件的第几页"。P5 回答的是"AI 正确的回答应该写成什么样"。不同类型的样本，标注方式不同：

**类型一：objective_qa（正常问答）**

AI 需要给一个确切的答案，评分分三层：

```
base_answer:  "宏挚传承2年缴付期，第2个保单年度保证利息10%，保单货币为美元"
               ↑ 核心事实，这句话错一个字 → 零分

required_keywords: ["宏挚传承", "2年缴", "保证利息"]
               ↑ 必须出现的实体，出现加分，不出现扣分

acceptable_answers: ["宏挚传承2年交，可享10%保证利息",
                     "2年缴付期的宏挚传承，保证利息为10%"]
               ↑ 语义一样的其他写法，算相似度

forbidden_content: ["6%", "6.5%", "找不到", "无相关信息"]
               ↑ 绝对不能出现的内容。6%/6.5%是3年缴的数据，答这个等于编造
```

**forbidden_content 不是凭空想的。** P4 召回了一些"分数高但不相关"的 chunk（比如 3 年缴的文档，里面写着 6%、6.5%），这些正是模型最容易错误引用的内容。标注员对着 P4 传过来的列表提炼 forbidden_content，比闭门造车准得多。

**类型二：refusal（正确拒答）**

AI 应该拒答，但拒答也有好坏之分：

```
forbidden_content: 不能编造的内容

expected_fallback: "建议查阅产品条款，或联系宏利客服"
               ↑ ★新增字段：拒答后该给什么兜底引导
               "我不知道" → 低分
               "我没有这个数据，建议您联系 XX" → 高分
```

**类型三：multi_point（综合多文档）**

AI 需要跨多份文档综合回答：

```
key_points: [
  "要点1：宏利支持第二受保人设置",
  "要点2：友邦需要投保人本人申请",
  "要点3：安达年龄限制为 55-85 岁"
]
```

每个要点独立计分，算覆盖率。

**类型四：clarify（需要反问澄清）**

问题信息不全，AI 应该反问：

```
clarify_question: "请问您指的是哪个缴付年期？2年缴还是5年缴？"
```

### ✅ P6 — Schema 校验

**做什么**：逐条检查 160 条样本是否符合 schema.json 的所有格式约束。

**输入**：P5 的完整标注集

**输出**：校验报告（哪条样本有什么格式错误）

**通俗理解**：

P6 是管线的最后一道自动检查。前面 P0-P5 混了自动填充和人工填写，任何一个环节都可能产出格式错误的数据。P6 逐条逐字段检查：

| 检查项 | 规则 | 示例错误 |
|---|---|---|
| 必填字段 | 9 个字段缺一不可 | `expected_behavior` 字段忘了填 |
| 枚举值 | 类型字段只能取白名单里的值 | `sample_type` 写了 "term"（不在白名单里） |
| hash 格式 | 必须是 40 位小写 hex | 写了 "abc123" |
| 页码 | 必须 ≥ 1 | 写了 page=0 |
| 条件约束 | 不同题型对应不同的必填字段 | refuse 题型填了 base_answer → 不允许 |

全部通过 = 格式层面没毛病了，可以进入 P7 双人核验。

## 关键设计决策

### 为什么不把拒答直接判死，而是留"待核验"？

企微真实数据中，系统说"找不到"不一定是真找不到。TERM-0007 就是例子——"预存利率"是口语，条款里叫"保证利息"，库里文档充分（score=0.78），系统拒答是术语未对齐导致的漏答，而非合理拒答。P1 先标"待核验"，P4 去库里查一圈后才最终判定。

### 为什么编号不用 OBJ/REF/MUL 前缀？

因为样本类型可能在 P4 被翻转。一条"疑似拒答"在 P4 发现库里其实有答案 → 翻转为 normal_answer。如果编号叫 REF-0001 但内容已经是 normal_answer，就名实不符。纯流水号 EVAL-0001 不论类型怎么变，ID 都不变。

### 为什么 gold_evidence 不用 document_id？

实测发现库里 `document_id` 恒为空字符串。而 `hash` 是文件级主键，跨库稳定，能直接定位到 OSS 上的原始文件。所以 gold_evidence 用 `hash + page` 来定位。

### 为什么 forbidden_content 要回看 P4 的检索结果？

标注员很难凭空穷举"模型可能编造什么"。但 P4 检索时发现的"高分但不相关"的 chunk，恰恰是模型在生成时最容易错误引用的。把这些 chunk 里的错误数值/结论写进 forbidden_content，是最准的标注方式。

| 术语 | 含义 |
|---|---|
| session | 一段完整企微聊天记录 |
| question unit | 从 session 中切出的一个独立问题 |
| sample_type | 样本的五大分类（normal_answer/refusal/multi_doc/chitchat/system_error） |
| gold_evidence | 人工确认的"标准答案出处"— 哪个文件(hash)的第几页(page) |
| base_answer | 人工写的"标准答案"—— AI 应该答出的核心事实 |
| forbidden_content | 明确不能出现的内容——出现即判模型幻觉 |
| DashVector | 阿里云向量数据库，存储保险文档的向量化索引 |
| OSS | 阿里云对象存储，存储保险文档原始文件 |
| hash | 文件的 SHA-1 哈希值，40 位 hex，作为文件级主键 |
| needs_term_mapping | 口语→条款术语映射标记（如"预存利率"→"保证利息"） |

## 环境依赖

- Python 3.12+
- DashVector SDK（P4 需要）
- 环境变量：`DASHSVECTOR_API_KEY`、`DASHSVECTOR_ENDPOINT`（P4 需要）

---

## 数据说明

- `real_conversation_recording/` 目录包含用户真实聊天记录，含姓名/手机/微信 ID 等隐私信息，**已加入 .gitignore 不提交**
- `intermediate/` 目录下的中间产物不含隐私信息，可以安全提交
- 知识库版本基准：`rag_v1_260604_bge_m3`
